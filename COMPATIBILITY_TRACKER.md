# SDK Compatibility Tracker

This document tracks where the public scripting SDK aligns with, intentionally abstracts, or diverges from Guida's current scripting behavior. It is public-facing: record observable compatibility behavior and adapter impact here, not private implementation details.

## Status Values

- `Aligned`: The SDK behavior matches the compatibility baseline.
- `Intentional Abstraction`: The SDK differs deliberately to keep the public package host-neutral.
- `Compatibility Risk`: The SDK shape may require adapter work or may change existing host behavior.
- `Deferred Slice`: The behavior is expected later in the extraction roadmap.
- `Needs Decision`: The intended compatibility policy is not settled yet.

## Divergence Register

| Area | SDK Behavior | Guida Compatibility Baseline | Divergence | Impact | Disposition |
| --- | --- | --- | --- | --- | --- |
| Task lifecycle start shape | `ScriptTaskManager.Start` returns an immediate `ScriptTaskHandle` with the running snapshot and completion task; `StartAsync` remains as a final-record convenience wrapper. | Hosts can start work, receive an immediate task id, observe running state, and later observe completion. | The SDK now supports immediate task tracking while preserving the await-final convenience API. | Adapters can keep host/UI task lists in sync without wrapping SDK execution themselves. | `Aligned` |
| Task status model | SDK exposes `Running`, `Completed`, `Failed`, `Canceled`, and `TimedOut`. | Existing host behavior distinguishes running, completed, failed, and canceled work; timeout behavior may be represented through cancellation or error paths. | SDK makes timeout a first-class terminal status and uses .NET `Canceled` spelling. | Adapters need explicit status mapping for host displays and serialized task results. | `Intentional Abstraction` |
| Task origins | SDK exposes generic origins plus public integration origins for MCP and intercepted host activity. | Existing host behavior distinguishes user, host, worker, queue, event, system, MCP, interception, and external work. | Public origins now cover observable integration categories while keeping private task metadata out of the core. | Origin-sensitive policy can key off public origins; tool names, queue item ids, worker indexes, trigger patterns, and intercepted-request details remain host-owned. | `Intentional Abstraction` |
| Execution policy metadata | SDK carries execution policy metadata in `ScriptExecutionInfo`, including access level and secret-access flags. | Existing host behavior distinguishes trusted work from restricted MCP/external execution for sensitive operations. | Policy is explicit metadata rather than implicit origin checks inside every capability. | Future script-facing API adapters can enforce raw-secret and secret-header behavior consistently without adding host-specific origin branches. | `Aligned` |
| Task metadata | `ScriptTaskRecord` stores generic lifecycle data: id, name, origin, language, timestamps, duration, error, return values, timeout, and script name. | Hosts may attach queue, worker, event, external-tool, trigger, and UI log metadata to task records. | Host-specific task metadata is not in the public core. | Hosts should keep private metadata in their own task registry or sidecar mapping unless a public pattern becomes necessary. | `Intentional Abstraction` |
| Task logs | SDK does not include per-task live log buffers. | Hosts may show recent task logs in UI surfaces. | Logging is represented separately through host capabilities, not embedded in task records. | UI adapters need to join log events to task ids through `ScriptExecutionInfo`. | `Intentional Abstraction` |
| Host context | SDK uses immutable `ScriptHostContext` plus explicit capabilities. | Existing host behavior uses a rich per-execution context containing many host services. | The SDK replaces service-bag coupling with public capability contracts. | Integration requires adapter objects for each public capability rather than exposing host services directly. | `Intentional Abstraction` |
| Capability coverage | SDK currently has logging, document loading, workspace, HTTP, secrets-related primitives, store primitives, queue primitives, worker primitives, and search primitives. | Existing scripts use broader API families, including storage, queues, search, workers, workflows, LLM, browser-shaped APIs, and host-specific UI or automation hooks. | Many capability families are not extracted yet. | Existing scripts cannot fully run through the SDK until more capability contracts and adapters exist. | `Deferred Slice` |
| Store capability | SDK exposes a host-neutral key-value store capability over opaque byte values, with structured result and error types. | Existing script storage behavior is host-backed and script-facing adapters decide object/string serialization. | SDK keeps persistence backend, authorization, workflow scoping, and language DTO mapping outside the core contract. | Hosts can adapt existing storage services behind `IScriptStore` while future language adapters define script-friendly value shapes. | `Intentional Abstraction` |
| Queue capability | SDK exposes a host-neutral queue capability over opaque byte payloads, with enqueue, claim, complete, abandon, get, and list operations. | Existing queue behavior is host-backed and can use custom dequeue strategies. | SDK does not depend on any concrete queue package and does not prescribe FIFO ordering. | Hosts can adapt strategy-capable queue implementations behind `IScriptQueue`; durable storage, retry policy, worker execution, and workflow ownership remain host concerns. | `Intentional Abstraction` |
| Worker capability | SDK exposes a host-neutral worker capability for submitting and tracking host-managed jobs. | Existing worker behavior is host-backed and may connect to queues, script tasks, and private worker pools. | SDK models worker job metadata and state transitions without owning scheduling, execution, retries, or queue consumption. | Hosts can adapt existing worker systems behind `IScriptWorker` and optionally link jobs to task ids; execution strategy and worker pools remain host concerns. | `Intentional Abstraction` |
| Worker script API metadata | The public registry preserves the existing `g.workers`, `g.worker`, and `g.worker.workflow` method names and key option/context fields for extracted worker surfaces. | Existing worker scripts use queue-worker management APIs and may receive host-specific execution context fields. | Browser-adjacent context and option fields are metadata only; the SDK core still does not implement browser tabs, layouts, or a worker pool scheduler. | Guida can keep its existing script-facing worker shape while non-browser hosts can omit or adapt those fields before exposing generated definitions. | `Intentional Abstraction` |
| Workflow ledger capability | SDK exposes a host-neutral workflow ledger capability with run, item, event, artifact, claim, retry, release, completion, failure, cancellation, dead-letter, bulk mutation, and schema validation/provider contracts. | Existing workflow behavior centers on a workspace-scoped durable ledger with string statuses/stages/states, item leases, retry/dead-letter handling, idempotent events, artifact dedupe, and schema-aware transitions. | SDK preserves the observable ledger contract while keeping durable persistence host-owned and BCL-only. Workspace schema-file loading is handled by workflow workspace discovery rather than the bare ledger core. | Guida can adapt its durable repository behind `IScriptWorkflowLedger`; script-facing adapters still need to project this contract into language-specific APIs. | `Aligned` |
| Workflow ledger administration and read models | SDK exposes `IScriptWorkflowLedgerAdministration` for retention preview/prune, history export/import, aggregate overview, transition graph, and flow evidence. The in-memory ledger implements the capability. | Existing ledger administration supports pruning old terminal run/item history, exporting and importing portable history payloads, dashboard-style counts and attention items, observed transition graphs with schema overlays, and queue flow evidence from event metadata. | SDK keeps durable transactions, SQLite vacuum implementation details, UI display models, MCP formatting, and file dialogs host-owned. Export/import is model-level and persistence-neutral. | Guida can adapt its durable ledger admin operations to the public administration capability while keeping storage and presentation private. | `Aligned` |
| Workflow queue/worker bridge helpers | SDK includes host-neutral helpers that upsert workflow items, enqueue JSON workflow envelopes onto `IScriptQueue`, append stable idempotent enqueue events, dispatch queue items to `IScriptWorker`, and create worker workflow contexts that prefer script task ids as ledger lease owners. | Existing workflow behavior connects ledger items to queued work and worker execution, with enqueue idempotency and task-id-aware workflow item completion/failure. | SDK keeps queue persistence, dequeue strategy, worker scheduling, and script execution host-owned; the bridge only composes public ledger, queue, and worker contracts. | Guida can keep its durable queue and worker execution model while adapting the observable workflow enqueue and worker-item semantics through the bridge helpers. | `Aligned` |
| Workflow workspace layout and discovery | SDK exposes `IScriptWorkflowWorkspaceDiscovery` plus a default `IScriptWorkspace`-backed discovery implementation for root `scripts/`, `lib/`, `views/`, `events.json`, `queues.json`, `tools.json`, workflow folders under `workflows/`, workflow manifests, workflow scripts/lib/views/config files, `workflow-ledger.schema.json` loading into the ledger schema validator, active-workflow overlay snapshots, and logical module/config path resolution helpers. | Existing workspace behavior uses the same root folders and workflow subfolder overlays; workflow manifests default to enabled, schema-only view JSON files are not counted as queue views, workflow views override global views by file name, config consumers load global plus workflow tiers, Lua probes active workflow scripts/lib before global lib/scripts, JavaScript module imports resolve from importer directory before workspace root, and workflow schema files are loaded per workflow. | SDK returns logical workspace paths and keeps all filesystem access and sandbox policy behind `IScriptWorkspace`. It does not create, rename, delete, import, export, switch, or activate workflows in this slice. | Guida can adapt the discovery surface to existing workspace folders without exposing private filesystem services; switching and script-facing `g.workflows` adapters remain later work. | `Aligned` |
| Workflow workspace management and read models | SDK exposes `IScriptWorkflowWorkspaceManager` for listing, summarizing, inspecting, activating/deactivating, creating, enabling/disabling, and portable import/export of workflow folders through `IScriptWorkspace`. | Existing workflow management lists workflows with enabled/status flags, tracks an active workflow, rejects disabled workflow activation in host tools, toggles enablement through `workflow.json`, and imports/exports workflow folders. | SDK keeps UI refresh, dropdown state, filesystem delete/rename, and zip-file packaging host-owned. Portable exports are model payloads rather than a prescribed archive format. | Guida can map its workflow UI/tools to the public manager while keeping destructive filesystem operations and archive dialogs in host code. Script-facing `g.workflows` adapters can project the manager read models later. | `Aligned` |
| Workflow ledger schema support | SDK includes schema-backed transition validation and schema-provider interfaces, including invalid-schema reporting. | Existing workflow schemas validate creation and transitions and provide schema information for graph/diagnostic overlays. | SDK parses and validates schema content but does not load `workflow-ledger.schema.json` from workspace folders in the ledger core. | Hosts can supply schemas from their workspace layer while the core ledger remains decoupled from filesystem layout. | `Intentional Abstraction` |
| Search capability | SDK exposes a host-neutral search capability with request, response, result item, and structured error types. | Existing search behavior is host-backed and may use app indexes, workspace indexes, web providers, vector search, or other private providers. | SDK defines the search boundary but does not implement indexing, ranking policy, provider choice, permissions, web search, or result hydration. | Hosts can adapt existing search providers behind `IScriptSearch`; language adapters can later map the compact result shape to script-facing DTOs. | `Intentional Abstraction` |
| Missing capabilities | Missing capability calls can be modeled through explicit unavailable-capability results. | Existing host behavior often depends on whether a service is present in the execution context. | SDK has a cleaner error-reporting pattern but not all adapters exist. | Future script API adapters should fail with clear capability errors instead of null or host-specific exceptions. | `Aligned` |
| Workspace core | SDK provides a sandboxed logical workspace with read, write, list, and entry metadata operations. | Guida workspaces have established folders and workflow-scoped overlays for scripts, libraries, views, and config files. | The SDK models safe file access separately from workflow layout conventions. | Integration can use the generic workspace directly or the workflow discovery helper for Guida-compatible layout conventions. | `Aligned` |
| Workspace path resolution | SDK direct workspace APIs reject unsafe paths and traversal, while `ScriptWorkspacePath.ResolveRelative` safely resolves adapter/module paths against a logical base directory. | Existing host behavior resolves relative paths from a base location and then checks that the final path remains inside the workspace. | Direct workspace access stays strict; base-relative adapter resolution is explicit. | Future native module loaders and config resolvers can preserve relative import/config behavior without weakening read/write workspace APIs. | `Intentional Abstraction` |
| Reparse point handling | SDK filesystem workspace denies reparse points by default. | Existing workspace safety primarily depends on final resolved path staying inside the workspace. | SDK adds a stronger sandbox default. | This improves default safety but may block workspaces that intentionally use symlinks. Hosts can decide whether to relax this option. | `Intentional Abstraction` |
| Document loading | SDK has `IScriptDocumentProvider` and a workspace-backed provider for logical document ids. | Existing JavaScript, Lua, and Janet flows load modules through each engine's native mechanisms. | The SDK does not provide a generic module resolver or source concatenation layer. | Concrete engines should adapt native module loading to the workspace/document boundary without changing script syntax. | `Deferred Slice` |
| Include preprocessing | SDK deliberately has no `#include` or source concatenation support. | Some historical behavior supported include-style preprocessing. | Include support is intentionally not carried forward. | Existing scripts that depend on includes need migration to native module syntax. | `Intentional Abstraction` |
| Pragmas | SDK does not parse comment pragmas such as timeout or required-context declarations. | Existing behavior may infer execution policy from script comments. | SDK prefers explicit execution options and capability availability. | Hosts should pass timeout/context policy explicitly when starting tasks. Ergonomics may need revisiting later. | `Intentional Abstraction` |
| JavaScript modules | SDK contracts are engine-neutral and do not yet implement JavaScript module loading. | Existing JavaScript scripts can use standard import syntax and debugging with URI-backed script files. | Concrete ClearScript integration is not extracted yet. | Phase 4 should preserve standard import syntax and file-backed debugging behavior. | `Deferred Slice` |
| Lua modules | SDK contracts are engine-neutral and do not yet implement Lua module loading. | Existing Lua scripts can use native module loading patterns. | Concrete Lua integration is not extracted yet. | Phase 4 should preserve native Lua module behavior rather than adding SDK-level preprocessing. | `Deferred Slice` |
| Janet modules | SDK contracts are engine-neutral and do not yet implement Janet module loading. | Existing Janet scripts can use native module import patterns. | Concrete Janet integration is not extracted yet. | Phase 4 should preserve native Janet module behavior rather than adding SDK-level preprocessing. | `Deferred Slice` |
| HTTP abstraction | SDK exposes `IScriptHttpClient` over `HttpRequestMessage` and `HttpResponseMessage`, with result/error wrappers. | Script-facing HTTP uses simple option and response objects. | SDK uses BCL HTTP types at the host boundary, not script-facing DTOs. | Script API adapters must translate script DTOs to BCL requests and back. | `Intentional Abstraction` |
| HTTP policy defaults | SDK `ScriptHttpPolicy` defaults to HTTPS-only and provides explicit `HttpsOnly` and `HttpAndHttps` presets. | Existing script HTTP behavior permits both HTTP and HTTPS when host policy allows it. | SDK keeps the secure public default and exposes a first-class compatibility preset for hosts that intentionally allow plain HTTP. | Guida-compatible hosts can opt into `HttpAndHttps` without hand-rolling scheme lists, while new SDK consumers remain HTTPS-only by default. | `Intentional Abstraction` |
| HTTP response sizing | SDK can enforce response body size when content length is known. | Existing script HTTP behavior returns response bodies as strings. | SDK adds a host policy boundary not visible in simple script DTOs. | Hosts need clear error mapping so policy failures are understandable to scripts. | `Intentional Abstraction` |
| HTTP secret headers | SDK supports host-side secret-backed header bindings on HTTP requests. | Existing tool execution can inject secret-backed HTTP headers without exposing secret values to external callers. | The SDK uses structured bindings rather than template strings as the public host boundary. | Adapters should parse host config/templates into `ScriptHttpSecretHeaderBinding` values before sending. | `Aligned` |
| Script-facing secrets | SDK exposes a host-side `IScriptSecretProvider`; execution policy now records whether raw secret access is allowed. | Existing scripts may retrieve raw secret values directly, with origin-specific access controls. | SDK intentionally avoids a public raw-secret script API for now. | The enforcement hook exists, but adding any raw script-facing secret API remains an explicit future decision. | `Needs Decision` |
| Secret error behavior | SDK models invalid names, not found, and access denied as structured secret errors. | Existing host behavior may throw or return language-specific nil/null/error shapes. | SDK uses structured errors at the host boundary. | Script language adapters must map these errors consistently per language. | `Intentional Abstraction` |
| Public API registry | SDK includes dependency-free descriptor contracts for script-facing types, functions, interfaces, API groups, registry documents, and registry providers. `ScriptApiKnownRegistries.CreateExtractedCapabilities()` publishes initial metadata for extracted store, queue, worker, workflow, workflows, and workspace surfaces. `ScriptApiTypeScriptGenerator` emits deterministic `.d.ts` text from the public registry. | Existing host tooling has a large static API metadata registry used for TypeScript definitions, completion metadata, and manifest/docs generation. | The SDK now covers public-safe TypeScript definition generation but not manifest, docs, Lua/Janet guide, editor completion, hover, or signature-help generation. It intentionally excludes private browser/UI/automation namespace groups. | Public adapters can start pinning script-facing names and generated TypeScript shapes for extracted capabilities. Remaining generator parity work is narrower and can target manifest/docs/completion outputs. | `Compatibility Risk` |
| Workspace script API metadata | SDK includes a public `g.workspace` descriptor for logical workspace file access: entry lookup, list, read, and write. | Existing compatibility-critical workspace behavior is mostly exposed through workflow discovery/switching and host tools rather than a broad general-purpose script namespace. | The low-level workspace descriptor is a new public SDK surface candidate, while `g.workflows` remains the compatibility target for workflow management scripts. | Hosts can choose whether to expose `g.workspace` to scripts, keep it host-only, or narrow it before TypeScript generation is finalized. This needs an explicit product/API decision before treating it as compatibility-locked. | `Needs Decision` |
| Concrete engines | SDK currently has engine contracts and a registration factory, but no JavaScript, Lua, or Janet engines. | Existing host behavior supports those languages. | Concrete engine implementation is pending. | Scripts cannot execute for real through the SDK without host-provided or future SDK engine registrations. | `Deferred Slice` |
| Engine factory behavior | SDK fails unknown or unregistered languages with clear errors inside task records. | Existing host behavior defaults unknown file names to JavaScript. | SDK is stricter when language cannot be determined or no engine is registered. | Hosts that want JavaScript-by-default should set the language explicitly before calling the SDK. | `Intentional Abstraction` |
| Dependency posture | SDK core remains dependency-light and BCL-oriented. | Existing host implementation uses concrete engine, desktop, browser, storage, and service dependencies. | Public core avoids host/runtime implementation dependencies. | Integration work should stay in adapters or engine packages, not the core package. | `Aligned` |

## Tracking Rules

- Keep this document public-safe: do not add private source paths, private class inventories, or closed implementation details.
- Record compatibility behavior, adapter impact, and decisions. Do not copy host implementation code or generated private API artifacts here.
- When a divergence is resolved, keep the row and update its disposition rather than deleting history.
- Add new rows when public contracts change in runtime, host capabilities, engine integration, workspace, HTTP, secrets, task lifecycle, API registry, queues, workers, workflows, or storage.
